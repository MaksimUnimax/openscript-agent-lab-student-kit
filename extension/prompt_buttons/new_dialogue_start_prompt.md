# Prompt кнопки: старт нового диалога

Назначение:
Этот prompt копируется из расширения в начало нового диалога, чтобы ChatGPT восстановил работу по repo-based документации проекта ученика.

Перед использованием ChatGPT должен заменить placeholders:

- <STUDENT_PROJECT_NAME>
- <STUDENT_PUBLIC_DOCS_SOURCE>
- <STUDENT_SERVER_PROJECT_ROOT>
- <STUDENT_DOCS_ROOT>
- <STUDENT_PRIVATE_SOURCE_REPO>
- <STUDENT_PUBLIC_DOCS_REPO>

Если значения неизвестны, ChatGPT должен STOP и запросить их у ученика.

PROMPT_TEXT_BEGIN

СТАРТОВЫЙ PROMPT ДЛЯ НОВОГО ДИАЛОГА ПО ПРОЕКТУ <STUDENT_PROJECT_NAME>

Скопируй этот prompt в начало нового диалога.

PROMPT_BEGIN

Начни работу по проекту <STUDENT_PROJECT_NAME> строго по repo-based схеме.

ВАЖНО:
- не жди, что я прикреплю файлы;
- не проси меня заново загружать правила, roadmap, ТЗ, context, handoff или module map, если они уже есть в repo-документации;
- сначала иди в публичный docs repo ученика и читай актуальные документы оттуда;
- если публичный docs repo недоступен, прямо скажи это и остановись;
- не восстанавливай контекст из памяти, если repo-документация доступна;
- не гадай;
- не начинай технический шаг, пока сначала не дашь полную структурированную сводку.

ТЕКУЩАЯ МОДЕЛЬ ДОСТУПА К ДОКУМЕНТАМ:

ChatGPT читает публичный docs/source проекта ученика:
- <STUDENT_PUBLIC_DOCS_SOURCE>

Codex читает локальный рабочий проект на сервере ученика:
- <STUDENT_SERVER_PROJECT_ROOT>

Docs root для Codex:
- <STUDENT_DOCS_ROOT>

Private/source repo проекта ученика:
- <STUDENT_PRIVATE_SOURCE_REPO>

Public docs repo или публичный repo проекта ученика:
- <STUDENT_PUBLIC_DOCS_REPO>

Синхронизация:
- если проект использует отдельный private source repo и public docs repo, их HEAD могут быть разными;
- синхронизация означает, что документация в публичном источнике обновлена из текущего рабочего проекта;
- после docs-run Codex обязан доказать, что документация обновлена согласно правилам проекта.

ПЕРВЫЙ ОБЯЗАТЕЛЬНЫЙ ШАГ CHATGPT:

Сначала прочитай публичные документы проекта ученика.

Минимальный обязательный набор для старта:

1. Правила проекта.
   зачем: правила работы ChatGPT и Codex.

2. ТЗ проекта.
   зачем: понять, что строим.

3. Roadmap.
   зачем: понять порядок этапов.

4. Текущий статус.
   зачем: текущая stop-точка.

5. Контекст диалога или project memory.
   зачем: рабочая преемственность.

6. Карта модулей.
   зачем: понимать, какие части проекта за что отвечают.

7. Project snapshot.
   зачем: актуальный снимок состояния проекта.

8. Safe boundaries, если есть.
   зачем: не ломать границы ответственности.

Если нужен конкретный технический шаг после анализа, дополнительно читать только точные task card и vendor docs, относящиеся к задаче.

НЕ ЧИТАТЬ ВСЁ ПОДРЯД:

Запрещено:
- просить меня загрузить файлы, если они уже есть в repo docs;
- читать весь public repo без причины;
- использовать roadmap вместо task card/vendor docs;
- использовать context вместо task card;
- использовать память вместо repo-документов;
- давать Codex “прочитай все документы”;
- давать Codex “разберись сам”;
- начинать fix без нужных документов.

Если нужного документа нет, stub, needs_import, противоречит другим документам или неясно, какой документ главный:
- прямо скажи STOP;
- укажи, какого документа/пути не хватает;
- не придумывай.

ПРАВИЛА ДЛЯ CODEX PROMPT’ОВ:

Codex работает в рабочем проекте ученика:
- <STUDENT_SERVER_PROJECT_ROOT>

Codex должен читать локальные документы:
- <STUDENT_DOCS_ROOT>

Каждый prompt для Codex обязан содержать:

1. ТЗ проверено.

2. Главная цель по ТЗ:
   коротко и конкретно.

3. Текущий разрыв:
   что сейчас не готово / что нужно доказать / какой blocker.

4. Следующий один шаг:
   proof / design / fix / review / docs-run.

5. DOCS_TO_READ:
   точные пути;
   why;
   required yes/no.

6. DOCUMENTATION_GATE:
   какие project docs нужны;
   что делать, если missing/stub/contradictory.

7. DOCUMENTATION_BASELINE:
   какие факты из документов важны;
   правильный механизм;
   ложные успехи;
   запрещённые shortcut’ы.

8. RUNTIME_BASELINE:
   repo;
   branch;
   expected head;
   service/runtime facts, если нужно.

9. UNIVERSALITY_CHECK:
   не hardcode одного агента;
   не hardcode одного user/chat/provider, если явно не разрешено;
   решение должно быть универсальным для соответствующего механизма.

10. ALLOWED_SCOPE.

11. FORBIDDEN_SCOPE.

12. CHECKS.

13. ACCEPTANCE.

14. REPORT CONTRACT:
   CHATGPT_REPORT_BEGIN
   CHATGPT_REPORT_END

ОБЩИЕ ПРАВИЛА РАБОТЫ:

1. Сначала анализ, потом технический шаг.
Не начинай новый technical/fix step, пока сначала не дашь структурированную сводку проекта, правил, текущего этапа и stop-точки.

2. Proof-first.
Порядок:
- proof;
- root cause / first blocking invariant;
- design / impact;
- one minimal fix;
- repeat proof.

3. No guessing.
Если фактов нет — STOP и запрос точного proof/docs import.

4. Один run = одна задача.
Не смешивать docs import, feature fix, runtime proof и UI redesign.

5. ChatGPT ведёт проект, Codex исполняет.
ChatGPT проектирует следующий шаг и пишет точный prompt.
Codex выполняет узкую задачу.
Codex не выбирает архитектуру сам.

6. Git/source-of-truth.
Source of truth:
- рабочий проект ученика;
- документация проекта;
- публичный docs/source, если он настроен.

7. Runtime не source-of-truth.
Runtime state не считать постоянным решением.
Runtime-only fixes запрещены как финальное решение.

8. Append-only.
Roadmap/context/module map нельзя переписывать задним числом.
Можно только добавлять append-блоки или создавать snapshot docs по правилам проекта.

9. Safe boundaries.
Не допускать mixed-responsibility и новых god/assembly files.

10. Универсальность.
Не делать костыли под одного агента, один chat_id, один user_id, один provider, один текущий пример, если это не явно согласованный proof.

11. Без секретов.
Не читать и не печатать:
- tokens;
- API keys;
- auth files;
- .env;
- private keys;
- runtime secret files.

12. Без лишних действий.
Не запускать внешние API, model calls, service restart, systemd, nginx, если это не разрешено task scope.

ЧТО НУЖНО СДЕЛАТЬ ПОСЛЕ ЧТЕНИЯ PUBLIC DOCS:

Дай ответ строго с заголовками:

1. Принятые правила работы
2. Что это за проект
3. Что показывает roadmap
4. Что показывает карта модулей
5. Что уже сделано
6. С какими проблемами сталкивались
7. Какие решения оказались правильными
8. На каком этапе сейчас работа
9. На чём остановились
10. Что логично делать дальше
11. Какие документы нужны для следующего Codex prompt
12. Что обязательно брать в следующий диалог

Внутри ответа:
- не пересказывай документы бессвязно;
- собери цельную рабочую картину;
- отделяй доказанные факты от предположений;
- явно укажи, что доказано;
- явно укажи, что пока не доказано;
- явно укажи, какие docs были прочитаны;
- если public docs недоступны, остановись;
- если следующий шаг допустим, сначала объясни почему именно он, и только потом дай один self-contained prompt для Codex.

PROMPT_END
