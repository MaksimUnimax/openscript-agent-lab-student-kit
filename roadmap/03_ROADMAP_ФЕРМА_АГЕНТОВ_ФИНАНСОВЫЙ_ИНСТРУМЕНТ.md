# 03. Roadmap: ферма агентов + Telegram + финансовый инструмент

Версия: `v0.1`  
Назначение: учебный roadmap для ChatGPT-наставника.  
Аудитория документа: ChatGPT, который ведёт ученика с абсолютного нуля.  
Ученик: человек без знаний программирования, умеет открыть браузер, копировать текст, вставлять текст и присылать отчёты обратно.

Этот документ не является классическим учебником программирования. Это маршрут, по которому ChatGPT ведёт ученика через создание рабочего проекта:

```text
ChatGPT-наставник → Codex-исполнитель → сервер → git → проект → проверка
```

# 0. Главная цель roadmap

Ученик должен собрать рабочий проект:

```text
Ферма агентов + Telegram-общение + финансовый инструмент + текст/чек/голос + запись в БД
```

К концу проекта у ученика должно быть:

1. свой сервер;
2. свой git-проект;
3. документация проекта внутри git;
4. визуальная страница структуры проекта;
5. основа Agent Lab;
6. Hermes как обязательный runtime для агентов;
7. фабрика agent package;
8. UI для документов агента;
9. первый агент, который отвечает через Hermes;
10. Telegram как первый внешний канал общения;
11. tool registry;
12. финансовый инструмент;
13. текстовый расход через Telegram с записью в БД;
14. фото чека → черновик → подтверждение → запись в БД;
15. голос → текст → тот же путь, что обычное сообщение;
16. второй агент;
17. финальная проверка;
18. обновлённая документация, чтобы ученик мог продолжать.

# 1. Жёсткие правила roadmap

## 1.1. Один шаг — одна задача

Нельзя смешивать в одном шаге:

```text
git + Telegram + Hermes + финансовый инструмент + голос + чек
```

Каждый маленький шаг должен иметь:

1. простое объяснение для ученика;
2. одну цель;
3. один prompt для Codex, если Codex нужен;
4. отчёт Codex;
5. проверку ChatGPT;
6. критерий “готово / не готово”.

## 1.2. Никакого localhost для ученика

Если ученик должен открыть страницу в браузере, ChatGPT не имеет права давать:

```text
localhost
127.0.0.1
IP тестового сервера другого человека
```

Только:

```text
http://<SERVER_PUBLIC_IP>/<PATH>
http://<SERVER_PUBLIC_IP>:<PORT>/<PATH>
https://<STUDENT_DOMAIN>/<PATH>
```

Для страницы структуры проекта универсальный адрес:

```text
http://<SERVER_PUBLIC_IP>/project-structure/
```

Если используется порт:

```text
http://<SERVER_PUBLIC_IP>:<PORT>/project-structure/
```

## 1.3. Hermes обязателен до настоящих агентов

Главная ошибка, которую нельзя повторять:

```text
Codex сделал агента, который отвечает не через Hermes.
```

В этом roadmap агент считается рабочим только если доказано:

```text
agent package → apply в Hermes runtime profile → reply через Hermes
```

Запрещено считать успехом:

```text
backend сам ответил за агента
fake/template reply выдан как работа агента
provider вызван напрямую мимо Hermes
один агент захардкожен в коде
документы сохранены, но Hermes profile не создан
```

Hermes — обязательный runtime-слой агента. Provider/model — источник ответа, который Hermes может использовать. Codex как исполнитель разработки — отдельная сущность.

Правильное объяснение ученику:

```text
Документы задают поведение агента.
Hermes запускает агента.
Backend только связывает части проекта и не должен подменять агента.
```

## 1.4. Telegram — канал общения, не бизнес-инструмент

Telegram идёт рано, потому что даёт ученику вау-эффект:

```text
Я написал агенту в Telegram — агент ответил.
```

Но Telegram нельзя путать с tool registry.

Правильное объяснение:

```text
Telegram — канал общения.
Финансовый модуль — первый бизнес-инструмент.
```

Telegram отвечает за доставку сообщений. Финансовый инструмент отвечает за действие: записать расход, создать черновик чека, подтвердить чек.

## 1.5. Финансовый инструмент — первый бизнес-tool

Финансовый инструмент появляется после того, как уже есть:

1. Hermes foundation;
2. agent package foundation;
3. UI фабрики агентов;
4. первый агент через Hermes;
5. Telegram-общение с агентом;
6. tool registry.

Финансовый слой должен быть deterministic: LLM/Hermes объясняет и ведёт диалог, но не пишет финансовые данные напрямую.

Правильная модель:

```text
Агент понял задачу → вызвал финансовый tool → tool записал данные → агент объяснил результат
```

## 1.6. Документы агента не являются рабочим агентом

Сохранить документы агента в UI — это ещё не значит, что агент работает.

Правильная цепочка:

```text
документы сохранены → package валиден → apply в Hermes → runtime profile обновлён → reply через Hermes доказан
```

UI должен показывать статус:

```text
Документы сохранены, но агент ещё не применён в Hermes.
```

## 1.7. Runtime не source-of-truth

Source — это git-tracked проект. Runtime — это рабочее состояние сервера.

Нельзя чинить проект так, чтобы исправление осталось только в runtime.

Правило:

```text
Изменение считается постоянным только если оно есть в source и сохранено в git.
```

## 1.8. Документация обновляется по факту

Документация не обновляется “по памяти”.

Обновление документации нужно:

1. после крупного завершённого этапа;
2. перед сменой большого блока;
3. по счётчику каждые N шагов, если были значимые изменения;
4. перед передачей проекта ученику.

Значение по умолчанию:

```text
N = 10 шагов
```

Но ChatGPT должен оценивать, были ли значимые изменения.

# Этап 1. Старт: сервер, git, документация

Следующий этап: [Этап 2. Ранняя визуализация структуры проекта](#этап-2-ранняя-визуализация-структуры-проекта)

## Цель этапа

Создать рабочую основу проекта до любого кода продукта.

На этом этапе ученик должен получить:

```text
сервер → папка проекта → git → документация → первый commit
```

## Почему этот этап первый

Нельзя начинать разработку “в воздухе”. Если нет git и документации, ChatGPT и Codex быстро начнут работать по памяти, а это приведёт к ошибкам.

## Шаг 1.1. Ученик загружает два стартовых документа

Ученик загружает в ChatGPT:

```text
01_ИНСТРУКЦИЯ_ДЛЯ_CHATGPT.md
02_СТАРТ_ПРОЕКТА_GIT_ДОКУМЕНТАЦИЯ_СТРУКТУРА.md
```

ChatGPT проверяет, что понял режим:

```text
ученик нулевой
объяснять просто
один шаг за раз
Codex работает на сервере
не давать localhost
```

Готово, если ChatGPT отвечает не лекцией, а первым рабочим шагом.

## Шаг 1.2. Проверить доступ к серверу

Цель: убедиться, что есть сервер, на котором будет жить проект.

ChatGPT объясняет:

```text
Сервер — это компьютер в интернете, где будет жить твой проект.
```

Codex должен проверить:

```text
кто пользователь
где рабочая папка
есть ли git
есть ли python
есть ли nginx/systemd при необходимости
какой публичный IP сервера
```

Готово, если есть подтверждённый:

```text
<SERVER_PUBLIC_IP>
```

## Шаг 1.3. Создать папку проекта

Цель: у проекта появляется отдельное место на сервере.

Пример универсального пути:

```text
/opt/openscript-agent-lab
```

Для ученика путь может быть другим, но должен быть единый project root.

Готово, если Codex сообщает:

```text
project_root:
exists: yes
```

## Шаг 1.4. Создать git

Цель: включить историю проекта.

ChatGPT объясняет:

```text
Git — это история проекта. Если что-то сломается, мы сможем понять, после какого изменения это произошло.
```

Готово, если:

```text
git repo initialized: yes
branch: main
first commit ready or created
```

## Шаг 1.5. Подключить общий публичный docs repo

Цель: не пересказывать документы вручную, а брать их из общего источника.

Общий repo нужен один для всех учеников. Codex должен брать его через HTTPS, без SSH.

Готово, если:

```text
public docs repo reachable: yes
docs copied or ready to copy: yes
```

## Шаг 1.6. Разложить стартовую документацию

В проект ученика должны попасть:

```text
docs/codex_source/
docs/codex_source/project/
docs/codex_source/roadmap/
docs/codex_source/rules/
docs/codex_source/module_map/
docs/codex_source/vendor/
docs/codex_source/training/
```

На старте можно положить только минимальный набор, но структура должна быть правильной.

Готово, если Codex показывает список документов и не кладёт секреты.

## Шаг 1.7. Первый commit

Цель: зафиксировать стартовое состояние.

Готово, если:

```text
git commit created: yes
git status clean: yes
```

Если есть unrelated dirty files — Codex обязан их перечислить и не трогать.

# Этап 2. Ранняя визуализация структуры проекта

Предыдущий этап: [Этап 1. Старт: сервер, git, документация](#этап-1-старт-сервер-git-документация)  
Следующий этап: [Этап 3. Общий каркас проекта](#этап-3-общий-каркас-проекта)

## Цель этапа

Ученик рано видит проект в браузере.

Это важный психологический и учебный момент: человек без опыта должен увидеть не абстрактный “код где-то на сервере”, а понятную структуру папок и файлов.

## Шаг 2.1. Создать безопасный backend для дерева проекта

Цель: получить дерево проекта только внутри project root.

Требования:

```text
не показывать содержимое файлов
не давать редактировать
не давать скачивать
не раскрывать абсолютные пути
не следовать symlink за пределы repo
скрывать .git, .env, token, auth, key, secret, password
```

Готово, если API возвращает только структуру:

```text
папка
файл
вложенность
тип
```

## Шаг 2.2. Создать standalone страницу

Страница:

```text
/project-structure/
```

Универсальный адрес для ученика:

```text
http://<SERVER_PUBLIC_IP>/project-structure/
```

Она не должна использовать основной UI Agent Lab.

Готово, если:

```text
uses_agent_lab_tabs: no
minimal_file_manager_style: yes
language_russian: yes
```

## Шаг 2.3. Проверить страницу по IP

ChatGPT обязан дать ученику только адрес его сервера:

```text
http://<SERVER_PUBLIC_IP>/project-structure/
```

Если Codex даёт `localhost`, шаг не принят.

Готово, если ученик открыл страницу и увидел папки.

## Шаг 2.4. Ручная проверка учеником

Ученик проверяет:

```text
страница открылась
видны папки
видны файлы
папки раскрываются
файлы не открываются
ничего нельзя редактировать
адрес не localhost
```

Готово, если ученик подтверждает глазами.

# Этап 3. Общий каркас проекта

Предыдущий этап: [Этап 2. Ранняя визуализация структуры проекта](#этап-2-ранняя-визуализация-структуры-проекта)  
Следующий этап: [Этап 4. Hermes foundation](#этап-4-hermes-foundation)

## Цель этапа

Создать минимальную основу Agent Lab до агентов, Telegram и финансов.

## Шаг 3.1. Создать базовую структуру папок

Минимум:

```text
agent_lab/
agent-packages/
docs/codex_source/
tests/
tools/
vendor/
```

Объяснение ученику:

```text
Сейчас мы создаём полки проекта. На одной будут исходники, на другой документы, на третьей агенты, на четвёртой проверки.
```

Готово, если папки видны в `/project-structure/`.

## Шаг 3.2. Разделить source и runtime

Source — то, что хранится в git.  
Runtime — рабочее состояние сервера.

Правило:

```text
source — источник правды
runtime — результат применения source
```

Готово, если в документации и коде есть явная граница.

## Шаг 3.3. Создать минимальный backend

Цель: проект умеет отвечать на HTTP-запросы.

Минимум:

```text
/healthz
```

Готово, если health check возвращает успешный ответ.

## Шаг 3.4. Создать минимальный static UI foundation

Это ещё не Agent Lab UI.

Минимум:

```text
простая страница
статус проекта
ссылка на /project-structure/
```

Готово, если страница открывается по серверному адресу.

## Шаг 3.5. Добавить первые tests

Минимум:

```text
test_health
test_project_structure_route
test_no_localhost_in_student_url
```

Готово, если тесты проходят.

## Шаг 3.6. Подключить systemd/nginx только минимально

Если нужен сервис, он должен запускать backend локально, а наружу отдавать через nginx.

Ученик не должен получать локальный адрес.

Готово, если есть:

```text
service active
public route works
health ok
```

## Шаг 3.7. Commit каркаса

Фиксируем каркас.

Готово, если:

```text
git commit created
git status clean except known unrelated
```

# Этап 4. Hermes foundation

Предыдущий этап: [Этап 3. Общий каркас проекта](#этап-3-общий-каркас-проекта)  
Следующий этап: [Этап 5. Agent package foundation](#этап-5-agent-package-foundation)

## Цель этапа

Встроить Hermes как обязательный runtime path до создания настоящих агентов.

Hermes — runtime-слой агента.  
Provider/model — источник ответа, который Hermes может использовать.  
Codex как исполнитель разработки — отдельная сущность.

## Шаг 4.1. Добавить Hermes как отдельный слой

Цель: проект знает, где Hermes, как проверить его наличие и как не смешивать Hermes с backend.

Готово, если:

```text
Hermes binary found or install plan exists
Hermes status check exists
backend does not replace Hermes
```

## Шаг 4.2. Ввести понятие Hermes runtime profile

Объяснение ученику:

```text
Документы агента лежат в проекте.
Рабочий профиль агента живёт в Hermes.
Сначала мы готовим документы, потом применяем их в Hermes.
```

Готово, если есть структура runtime profile и проверка статуса.

## Шаг 4.3. Создать Hermes status check

Цель: проверять readiness без чтения секретов.

Нельзя читать:

```text
auth.json
.env
tokens
API keys
```

Готово, если status показывает:

```text
profile exists yes/no
auth metadata yes/no
config present yes/no
secrets not printed
```

## Шаг 4.4. Создать dry/smoke path без live model call

Цель: доказать, что связка project → Hermes существует без реального запроса к модели.

Готово, если:

```text
dry proof passed
no provider/model call made
```

## Шаг 4.5. Запретить backend/template reply как успех

Это главный предохранитель.

В acceptance каждого агентного шага должно быть:

```text
reply_path_uses_hermes: yes/no
direct_backend_reply_bypass_detected: yes/no
fake_template_reply_used_as_success: yes/no
provider_called_directly_without_hermes: yes/no
```

Готово только если обходов нет.

## Шаг 4.6. Зафиксировать Hermes-only invariant в docs

Документ должен сказать:

```text
Агент не считается рабочим, пока не доказан reply через Hermes.
```

Готово, если это правило есть в roadmap/TЗ/prompt templates.

# Этап 5. Agent package foundation

Предыдущий этап: [Этап 4. Hermes foundation](#этап-4-hermes-foundation)  
Следующий этап: [Этап 6. UI фабрики агентов](#этап-6-ui-фабрики-агентов)

## Цель этапа

Создать формат агента, но пока не создавать первого учебного агента как продукт.

## Шаг 5.1. Описать полный agent package

Минимум:

```text
manifest.json
SOUL.md
rules.md
examples.md
README.md
provider.defaults.json
skills/
tools.json
```

Объяснение ученику:

```text
Агент — это не один текст. Это папка с документами, навыками, настройками и будущими инструментами.
```

## Шаг 5.2. Создать validator agent package

Validator проверяет:

```text
есть manifest.json
есть SOUL.md
есть rules.md
есть examples.md
есть README.md
есть provider.defaults.json
есть skills/
есть tools.json
нет запрещённых имён
нет пустого package
```

Готово, если invalid package не проходит.

## Шаг 5.3. Связать package и Hermes profile

Цель: agent package должен применяться в Hermes runtime profile.

Готово, если есть flow:

```text
source package → validate → apply → runtime profile
```

## Шаг 5.4. Добавить статус package

Статусы:

```text
draft
valid
not_applied
applied
runtime_mismatch
blocked
```

Готово, если UI/API может показать статус.

## Шаг 5.5. Добавить tools.json как обязательную часть

Даже если инструментов ещё нет, `tools.json` должен быть частью формата.

Почему:

```text
Агент сразу проектируется как сущность, которая может иметь инструменты.
```

Готово, если package без tools.json не считается полным.

## Шаг 5.6. Добавить skills/ как обязательную область

Навыки могут быть пустыми, но папка должна существовать.

Готово, если skills/ есть и отображается в структуре.

## Шаг 5.7. Smoke package только для проверки механизма

Разрешён технический fixture, но не учебный агент.

Условия:

```text
не показывать как первого агента
не использовать как Telegram-агента
не hardcode
использовать только для проверки validate/apply/status
```

Готово, если smoke package проверяет механизм, но не становится продуктовым агентом.

# Этап 7. Первый агент + Telegram

Создать первого настоящего агента и дать ученику поговорить с ним в Telegram.

Telegram здесь — первый внешний канал общения и вау-checkpoint, не бизнес-tool.

## Шаг 7.1. ChatGPT генерирует документы первого агента

Например агент:

```text
Учебный помощник OpenScript
```

Документы:


Готово, если package валиден.

## Шаг 7.3. Применить package в Hermes

Готово, если:

```text
runtime profile exists
package applied
status ok
```

## Шаг 7.4. Проверить reply через Hermes

Это ещё может быть не Telegram.

Готово только если:

```text
reply_path_uses_hermes: yes
direct_backend_reply_bypass_detected: no
fake_template_reply_used_as_success: no
```

## Шаг 7.5. Настроить Telegram connector

Цель: подключить Telegram как канал общения.

```text
bot token configured without printing secret
connector state ok
live mode known
```

Правило:

```text
from.id / user_id — доступ
```

Готово, если unknown user denied before expensive resource use.

## Шаг 7.7. Настроить routing selected agent

Цель: Telegram message должен попасть к выбранному агенту.

Готово, если:

```text
selected_agent resolves correctly
system agents not offered as normal candidates
```

## Шаг 7.8. Live Telegram proof

Ученик пишет агенту в Telegram.

Успех:

```text
сообщение получено
allowlist passed
selected agent chosen
reply through Hermes
answer delivered to Telegram
backend/template bypass no
```

## Шаг 7.9. Проверить финальное состояние Telegram

Нельзя принять успех, если после теста бот сломан.

Готово, если:

```text
polling alive
connector not paused
last reply delivered
final bot state working
```

# Этап 8. Tool registry

Предыдущий этап: [Этап 7. Первый агент + Telegram](#этап-7-первый-агент--telegram)  
Следующий этап: [Этап 9. Финансовый инструмент через Telegram](#этап-9-финансовый-инструмент-через-telegram)

## Цель этапа

Создать систему инструментов, которые агент может получать по разрешению.

## Шаг 8.1. Создать tool catalog

Каталог должен хранить:

```text
tool id
название
описание
статус ready/blocked
опасность
toolset
handler
```

## Шаг 8.2. Ввести blocked-by-default

Опасные tools по умолчанию заблокированы:

```text
browser
web
files
terminal
mcp
```

## Шаг 8.3. Ввести ready tools

Сначала готовим только безопасные и нужные.

Позже:

```text
fin.expense_add
fin.receipt
```

## Шаг 8.4. Привязать tools к agent package

Связь идёт через:

```text
tools.json
```

Готово, если agent package может иметь список разрешённых tools.

## Шаг 8.5. Применить tool binding в Hermes profile

Недостаточно сохранить tools.json.

Готово только если после apply Hermes видит tool schemas.

## Шаг 8.6. Provider-safe tool names

Внутренние имена вроде:

```text
fin.expense_add
```

могут быть непригодны для provider-visible schema.

Нужно:

```text
internal tool id → provider-safe name → обратно в internal tool id
```

Готово, если нет tool names с недопустимыми символами в provider request.

## Шаг 8.7. Проверить permissions

Разные агенты могут иметь разные tools.

Готово, если агент без financial tool не видит financial tool.

# Этап 9. Финансовый инструмент через Telegram

Предыдущий этап: [Этап 8. Tool registry](#этап-8-tool-registry)  
Следующий этап: [Этап 10. Фото, голос, второй агент, финал](#этап-10-фото-голос-второй-агент-финал)

## Цель этапа

Агент в Telegram не просто отвечает, а делает полезное действие через финансовый инструмент.

## Шаг 9.1. Создать финансовый storage root

Не писать в production DB во время proof.

Для ученика создаётся его runtime DB.

Готово, если DB path определён и не смешан с source.

## Шаг 9.2. Создать SQLite schema

Таблицы:

```text
schema_version
schema_migrations
categories
payment_sources
expense_records
receipt_drafts
receipt_item_lines
operation_log
audit_events
```

## Шаг 9.3. Создать storage initializer

Цель: БД можно создать повторяемо.

Готово, если initializer создаёт schema и базовые справочники.

## Шаг 9.4. Создать deterministic expense add

Flow:

```text
validate request
normalize amount
normalize category
normalize payment source
write expense_records
write operation_log/audit_events
return structured result
```

LLM не пишет SQL.

## Шаг 9.5. Создать recent expenses

Read-only операция.

Готово, если можно показать последние расходы.

## Шаг 9.6. Создать month summary

Read-only операция.

Готово, если можно получить сумму за месяц.

## Шаг 9.7. Подключить financial tool к Hermes

Tool должен быть виден Hermes только через registry/binding.

Готово, если Hermes sees schema and handler deterministic.

## Шаг 9.8. Привязать financial tool к агенту

Через `tools.json`.

Потом apply в Hermes.

## Шаг 9.9. Проверить текстовый расход через Telegram

Ученик пишет:

```text
Потратил 500 рублей на кофе
```

Успех:

```text
Telegram получил сообщение
agent selected
Hermes reply path
tool called
expense written to DB
agent replied
```

## Шаг 9.10. Проверить запись в БД

Нельзя принимать только текст ответа.

Нужно проверить:

```text
expense_records row exists
operation_log written
audit_events written
recent expenses shows it
```

## Шаг 9.11. Объяснить ученику boundary

Простое объяснение:

```text
Агент не сам записал расход. Агент попросил финансовый инструмент. Инструмент записал данные в базу и вернул агенту результат.
```

# 4. Примерное количество маленьких шагов

Оценка:

```text
Этап 1: 7 шагов
Этап 2: 4 шага
Этап 3: 7 шагов
Этап 4: 6 шагов
Этап 5: 7 шагов
Этап 6: 8 шагов
Этап 7: 9 шагов
Этап 8: 7 шагов
Этап 9: 11 шагов
Этап 10: 24 шага
```

Итого:

```text
90 маленьких шагов
```

Реальный диапазон:

```text
85–100 шагов
```

Почему не меньше:

1. ученик нулевой;
2. нельзя смешивать Telegram, Hermes, tools, БД, OCR и voice;
3. надо защищаться от старых ошибок;
4. каждый успех должен быть проверяемым;
5. главное обучение — это повторение цикла “задача → Codex → отчёт → проверка”.
